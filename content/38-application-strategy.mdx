## 들어가며

취준 커뮤니티에서 자주 보이는 두 가지 전략:

> A: "무조건 많이 넣어야 해. 나는 100개 넣었어."
> B: "맞춤형으로 10개만 집중했어."

**누가 먼저 붙었을까요?**

정답은 "그때그때 다르다"지만, 데이터를 보면 **어떤 전략이 효율적인지** 꽤 명확해집니다.

---

## 왜 "100개 뿌리기"는 비효율적인가

### 이유 1: 공고마다 원하는 게 다르다

101개 공고의 기술 조합을 보면:

| 공고 유형 | 요구 기술 | 비율 |
|-----------|----------|------|
| React 중심 | React + TypeScript | 40% |
| Next.js 포함 | React + TS + Next.js | 28% |
| 풀스택 성격 | React + Node.js + AWS | 10% |
| 모바일 포함 | React + React Native | 8% |
| 특수 (3D, 지도 등) | React + 도메인 기술 | 5% |

**같은 "프론트엔드" 공고인데 원하는 기술 조합이 다릅니다.** 하나의 이력서로 100개에 다 맞출 수 없어요.

### 이유 2: 맞춤형 이력서가 서류 통과율 압도

일반적인 서류 통과율:

| 전략 | 예상 서류 통과율 | 면접 기회 |
|------|----------------|----------|
| 100개 동일 이력서 | 5~10% | 5~10개 |
| 30개 약간 수정 | 15~20% | 5~6개 |
| 10개 맞춤형 | 30~40% | 3~4개 |

**면접 기회는 비슷하지만, 투입 시간은 10배 차이.**

그리고 맞춤형으로 넣으면 **"이 사람이 우리 공고를 제대로 읽었구나"**라는 인상을 줘요. 이게 서류 통과율을 크게 올립니다.

### 이유 3: 면접 준비 시간이 부족해진다

100개 넣으면 면접이 겹칩니다. 회사별 기술 스택 파악, 면접 질문 준비를 제대로 못 해요.

10개 넣으면 각 회사를 **깊이 조사**할 수 있습니다. 기술 블로그 읽고, 제품 써보고, 면접 질문 예상하고.

## 최적의 지원 전략: "3단계 깔때기"

### 1단계: 공고 50개 리서치 (2~3일)

원티드, 잡코리아, 사람인에서 "프론트엔드" 검색 후 50개 정도 훑어보세요.

**이 단계에서 하는 일**:
- 공고 읽기 (5분씩)
- 3가지로 분류:
  - A등급: "나한테 딱 맞다" (기술 80%+ 매칭)
  - B등급: "좀 맞는다" (기술 60~79% 매칭)
  - C등급: "안 맞는다" (기술 60% 미만)

### 2단계: A등급 10~15개 맞춤 지원 (1~2주)

A등급 공고에 **맞춤형 이력서**를 보냅니다.

**맞춤화 포인트**:
1. **프로젝트 순서 변경** — 공고에서 요구하는 기술을 쓴 프로젝트를 위로
2. **기술 키워드 매칭** — 공고에 "React Query"가 있으면 이력서에도 "React Query" 명시
3. **자기소개 1~2문장 수정** — 회사/포지션에 맞게

> 전부 새로 쓰라는 게 아닙니다. **기본 이력서 80% + 맞춤 20%**면 충분해요.

### 3단계: B등급 10~15개 표준 지원 (1주)

B등급은 기본 이력서로 보내되, **기술 스택만 약간 조정**.

A등급에서 면접 기회가 부족하면 B등급으로 넓히는 전략.

---

## 공고 101개로 보는 "맞춤형 지원" 실전

### 예시: React Query 경험이 있는 취준생

101개 중 React Query/TanStack을 자격요건에 넣은 공고: **11개 (11%)**

이 11개 공고에 집중 지원하면:
- 다른 지원자보다 **확실히 차별화**
- 이력서에 "React Query로 서버 상태 캐싱 구현" 명시
- 면접에서 React Query 관련 깊은 대화 가능

**"내가 가진 강점"과 "공고가 원하는 것"의 교집합**을 찾는 게 핵심입니다.

### 예시: Next.js 프로젝트가 있는 취준생

Next.js를 요구하는 공고: **28개 (28%)**

이 28개 중 "퇴사율 5% 이하 + 연봉 업계평균 이상"인 곳을 필터링하면 → 타겟 리스트 완성.

## 이력서 맞춤화 체크리스트

공고를 읽은 후, 이력서를 보내기 전에:

- [ ] 공고의 **자격요건 기술**과 내 이력서의 기술이 **매칭**되는가?
- [ ] 매칭되는 프로젝트가 이력서 **상단에** 있는가?
- [ ] 공고에서 쓴 **기술 용어**를 이력서에도 동일하게 사용했는가?
  - 공고: "React Query" → 이력서: "React Query" (○)
  - 공고: "React Query" → 이력서: "TanStack Query" (△ 같은 건데 매칭 안 될 수도)
- [ ] 공고의 **업무 내용**과 관련된 경험이 이력서에 드러나는가?
- [ ] **배포 링크**(Vercel/GitHub)가 포함되어 있는가?

## "넣을 곳이 없다"는 분에게

"10개 공고도 못 찾겠어요. 다 경력직이에요."

이전 편(C4)에서 분석했듯이:

1. **"경력 무관" 또는 미기재** 공고: 101개 중 27개 (27%)
2. **신입 직접 명시**: 12개 (12%)
3. **기술 3개 이하 요구**: 추가로 10~15개

합치면 **40~50개 정도 지원 가능한 공고**가 있어요.

그리고:
- 원티드만이 아니라 잡코리아, 사람인, 로켓펀치, 프로그래머스 채용도 확인
- LinkedIn 채용 (외국계)
- 회사 채용 페이지 직접 확인 (원티드에 안 올리는 곳도 많음)

소스를 다양화하면 선택지가 확 넓어집니다.

## 지원 시기 전략

### 공고 올라온 직후가 유리합니다

- 초반 지원자는 **꼼꼼히 검토**됨
- 후반 지원자는 이미 면접 진행 중이라 **빠르게 스크리닝**될 수 있음
- 원티드 알림 설정: "프론트엔드" 키워드로 새 공고 알림 받기

### 월요일~수요일 오전이 유리하다는 설

- HR팀이 주 초에 서류 검토하는 경향
- 금요일 저녁에 보낸 이력서는 월요일 아침 수십 개 속에 묻힘
- **과학적 근거는 약하지만**, 의식하면 나쁠 것 없음

## 정리

| 100개 뿌리기 | 10개 집중 |
|-------------|----------|
| 동일 이력서 | 공고별 맞춤 |
| 서류 통과 5~10% | 서류 통과 30~40% |
| 면접 준비 부실 | 회사별 깊은 준비 |
| 지원 = 양 | 지원 = 질 |
| 시간 대비 비효율 | 시간 대비 효율 |

**"많이 넣으면 하나는 걸리겠지"는 로또 전략입니다.** 이력서도 포트폴리오도 제한된 자원이에요. 전략적으로 집중하면 더 빠르게 합격합니다.
