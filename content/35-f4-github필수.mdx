## 들어가며

취준생 사이에서 "잔디 심기"가 유행입니다. 매일 커밋해서 GitHub 프로필의 초록색 칸을 채우는 거죠.

> "1일 1커밋 200일째! 면접에서 성실함을 어필할 수 있겠지?"

솔직히 말할게요. **면접관은 잔디를 안 봅니다.** 아니, 정확히는 잔디"만" 보고 판단하지 않아요.

101개 공고를 분석해보니, 17개(17%)가 GitHub을 직접 언급했는데 — **하나도 "매일 커밋하세요"라고 쓴 곳은 없었습니다.**

---

## 공고에서 GitHub은 어떻게 언급되나

17개 공고의 GitHub 언급 맥락:

| 맥락 | 예시 | 수 |
|------|------|-----|
| **협업 도구로서** | "Git/GitHub (PR 기반 협업)" | 10개 |
| **제출물로서** | "이력서 및 포트폴리오(또는 GitHub 링크) 제출" | 4개 |
| **기술 스택 나열** | "Tools: GitHub, Slack, Notion" | 3개 |

주목할 점: **"GitHub 활동 내역을 봅니다"라고 쓴 공고는 0개**입니다.

GitHub은 "코드를 보는 곳"이지, "출석부"가 아니에요.

---

## 면접관이 GitHub에서 실제로 보는 것

### 1순위: 코드 품질

```
🔍 면접관의 시선 경로:
프로필 → 대표 프로젝트 클릭 → README → src 폴더 → 코드 파일 2~3개
```

총 소요 시간: **2~5분**

이 짧은 시간에 판단하는 것:
- 파일/폴더 구조가 논리적인가
- 변수/함수 네이밍이 깔끔한가
- 컴포넌트가 적절히 분리되어 있는가
- TypeScript를 쓰고 있는가

### 2순위: 커밋 메시지

```
// 면접관이 한숨 쉬는 커밋 로그
"ㅋㅋ"
"asdf"
"update"
"fix fix fix"

// 면접관이 고개 끄덕이는 커밋 로그
"feat: 상품 검색 필터 컴포넌트 구현"
"fix: 무한 스크롤 중복 요청 방지 (debounce 적용)"
"refactor: API 호출 로직을 커스텀 훅으로 분리"
```

커밋 메시지는 **"이 사람이 팀에서 일하면 어떤 커밋을 남길까"**를 보여줍니다.

### 3순위: README

README가 없는 프로젝트 = **"나는 이 프로젝트에 대해 설명할 생각이 없습니다"**

README가 있는 프로젝트 = **"이 프로젝트의 목적, 기술, 고민을 정리할 줄 압니다"**

### 4순위 (보너스): PR과 이슈

혼자 작업하더라도 **브랜치를 나누고 PR을 만들면** 협업 경험이 있어 보입니다.

```
main ← feature/search-filter (PR #3)
main ← fix/infinite-scroll-bug (PR #4)
```

### 잔디: 기본은 된다 — 없으면 안 된다

오해하면 안 되는 게, **잔디 자체는 중요합니다.**

"개발을 좋아합니다"라고 말하면서 GitHub 잔디가 텅텅 비어있으면? **그 말을 믿기 어렵습니다.** 특히 신입은 실무 경력이 없으니까, 잔디는 "이 사람이 꾸준히 코딩하고 있구나"라는 **최소한의 신뢰**를 줍니다.

하지만 잔디만으로 합격하진 않아요. **잔디 = 기본, 코드/README/커밋 = 차별화**입니다.

---

## 커밋 메시지를 잘 쓸 자신이 없다면? → Private

```
"ㅋㅋ"
"asdf"
"update"
"일일 커밋: README 마침표 수정"
```

이런 커밋이 public 레포에 있으면, 면접관은 이렇게 생각합니다:

> "잔디 채우려고 억지로 커밋하는구나. 이 사람 팀에 오면 커밋 로그를 어지럽히겠다."

**해결책은 간단합니다.** 연습용, 학습용 레포는 **private**으로 만드세요. 잔디는 private 레포도 카운트됩니다. 잔디는 심되, 부끄러운 커밋은 숨기고, **보여줄 프로젝트만 public**으로 깔끔하게 관리하면 됩니다.

## GitHub 프로필 최적화 (30분이면 됨)

### Step 1: 대표 프로젝트 핀 고정 (5분)

GitHub 프로필에서 **가장 자신 있는 프로젝트 2~3개를 핀으로 고정**하세요. 면접관은 핀 된 프로젝트부터 봅니다.

핀 기준:
1. 가장 기술적으로 도전적인 프로젝트
2. 코드가 가장 깔끔한 프로젝트
3. README가 잘 쓰인 프로젝트

### Step 2: README 정리 (20분)

핀 된 프로젝트의 README에 **반드시 포함할 것**:
- 프로젝트 한 줄 설명
- 기술 스택 (왜 선택했는지 한 줄씩)
- 스크린샷 또는 배포 링크
- 실행 방법

### Step 3: 프로필 README (5분)

GitHub 프로필에 자기소개를 추가할 수 있어요. (username/username 레포 생성)

```markdown
## 안녕하세요, 프론트엔드 개발자 [이름]입니다

- 🔧 React + TypeScript + Next.js
- 📚 서버 상태(React Query)와 클라이언트 상태(Zustand) 분리에 관심
- 🌱 현재 테스트 코드 작성과 접근성 개선을 공부 중
```

간결하게. 이력서가 아니라 **첫인상**이니까요.

## 잔디 vs 실질 — 비교 실험

| | A씨 (잔디만) | B씨 (잔디 + 실질) |
|---|---|---|
| GitHub 프로필 | 365일 잔디 | 잔디 + 핀 프로젝트 |
| 레포 구성 | public 20개 (튜토리얼) | private 학습용 + public 3개 |
| README | 없거나 "TODO" | 기술 선택 이유 + 스크린샷 |
| 커밋 메시지 | "update", "fix" | Conventional Commits |
| 코드 품질 | 파일 1개에 500줄 | 역할별 분리, 타입 정의 |
| **면접관 인상** | "성실하긴 한데..." | "이 사람 코드 깔끔하네" |

**결과**: 둘 다 잔디는 있지만, B씨가 기술면접을 통과할 확률이 압도적으로 높습니다.

## 정리

| 기본 (잔디) | 차별화 (잔디 위에 쌓는 것) |
|-----------|----------|
| 꾸준히 커밋 (private OK) | 의미 있는 커밋 메시지 (public) |
| 잔디로 최소 신뢰 | 핀 고정 2~3개 + README |
| 학습용 레포 유지 | PR + 브랜치 전략 |
| "개발하고 있구나" | "이 사람 코드 깔끔하네" |

**잔디는 기본이고, 그 위에 코드 품질과 README가 차별화입니다.** 커밋 메시지를 잘 쓸 자신이 없으면 private으로 만들고, 보여줄 프로젝트만 public으로 깔끔하게. 면접관이 2분 안에 "이 사람 괜찮다"고 느끼게 만드는 게 목표입니다.
