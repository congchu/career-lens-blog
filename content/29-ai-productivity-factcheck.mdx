## 들어가며

AI 코딩 도구 광고를 보면 이런 문구가 넘칩니다.

> "개발 속도 10배 향상"
> "AI로 하루 만에 앱 완성"
> "코딩 없이 서비스 출시"

진짜일까요? **팩트체크 해봤습니다.**

결론부터: 빨라지는 건 맞습니다. 하지만 **"10배"는 과장이고, "빨라지는 영역"이 한정되어 있습니다.**

---

## 팩트 1: GitHub Copilot 실험 — 55% 더 빠르다

가장 유명한 데이터부터 보겠습니다. GitHub이 직접 실험한 결과예요.

### 실험 설계

- **참가자**: 전문 개발자 95명 (취미 개발자 아님)
- **과제**: JavaScript로 HTTP 서버 작성
- **방법**: 랜덤으로 두 그룹 나눔 (Copilot O / Copilot X)
- **측정**: 완료 시간 + 완료율

### 결과

| 항목 | Copilot 사용 | Copilot 미사용 |
|------|-------------|---------------|
| **평균 완료 시간** | 1시간 11분 | 2시간 41분 |
| **완료율** | 78% | 70% |
| **속도 차이** | **55% 더 빠름** | — |

통계적으로 유의미한 결과입니다 (P=0.0017). 95% 신뢰구간은 21%~89%.

> 📎 출처: [GitHub Blog — Research: quantifying GitHub Copilot's impact on developer productivity and happiness](https://github.blog/news-insights/research/research-quantifying-github-copilots-impact-on-developer-productivity-and-happiness/)

### 여기서 주의할 점

**55% 빨라진 건 "코드 작성" 속도입니다.** 개발의 전체 과정이 아니에요.

실제 개발 업무를 시간으로 나눠보면:

```
코드 작성     ████████░░░░░░░░░░░░  30~40%
디버깅/테스트  ██████░░░░░░░░░░░░░░  25~30%
설계/논의     █████░░░░░░░░░░░░░░░  20~25%
코드 리뷰     ███░░░░░░░░░░░░░░░░░  10~15%
문서화        █░░░░░░░░░░░░░░░░░░░   5%
```

AI가 직접 빠르게 해주는 건 "코드 작성" 부분이에요. 전체 개발 시간에서 30~40%를 차지하는 부분이 55% 빨라진 거니까, **실제 체감 속도 향상은 15~25% 정도**입니다.

"10배 빨라졌다"는 건 **아주 단순한 작업(보일러플레이트 생성 등)**에서만 성립하는 이야기예요.

## 팩트 2: Stack Overflow 설문 — 76%가 사용 중이거나 예정

Stack Overflow 2024 개발자 설문에서 AI 도구 관련 데이터:

| 항목 | 수치 |
|------|------|
| AI 도구 사용 중이거나 사용 예정 | **76%** (전년 70%에서 상승) |
| 현재 사용 중 | **62%** (전년 44%에서 급증) |
| AI 도구에 호의적 | **72%** (전년 77%에서 하락) |
| AI 출력 신뢰 | **43%** (31%는 회의적) |
| AI가 복잡한 작업을 잘 못한다 | **45%** |

> 📎 출처: [Stack Overflow 2024 Developer Survey — AI](https://survey.stackoverflow.co/2024/ai)

### 재미있는 포인트

**사용률은 올라갔는데, 호의도는 내려갔습니다** (77% → 72%).

이게 무슨 뜻이냐면, "써보니까 생각보다 완벽하지 않더라"는 겁니다. 기대가 높았는데 현실은 좀 다른 거죠.

특히 **"복잡한 작업을 잘 못한다"에 45%가 동의**한 건 중요해요. 투두앱 만들기 같은 간단한 작업은 AI가 잘하지만, 실무에서 만나는 복잡한 문제(레거시 코드 리팩토링, 성능 최적화, 다중 서비스 연동 등)에서는 한계가 명확하다는 뜻입니다.

## 팩트 3: "빨라지는 것" vs "안 빨라지는 것"

Stack Overflow와 GitHub 데이터를 종합하면, AI 코딩 도구가 실제로 도움이 되는 영역과 아닌 영역이 명확합니다.

### 확실히 빨라지는 것

| 작업 | AI 도움 정도 | 왜? |
|------|-------------|-----|
| 보일러플레이트 코드 작성 | 매우 높음 | 패턴이 명확, 반복적 |
| 테스트 코드 생성 | 높음 | 입출력만 알면 자동 생성 |
| 문서화/주석 | 매우 높음 | 코드 보고 요약하는 건 AI가 잘함 |
| 코드 변환 (JS→TS 등) | 높음 | 규칙 기반 변환 |
| API 연동 코드 | 높음 | 문서 보고 코드 만드는 패턴 |
| CSS/스타일링 | 높음 | 디자인→코드 변환 |

### 별로 안 빨라지는 것

| 작업 | AI 도움 정도 | 왜? |
|------|-------------|-----|
| 아키텍처 설계 | 낮음 | 프로젝트 맥락을 모름 |
| 복잡한 버그 디버깅 | 중간 | 환경/상태 의존적 |
| 성능 최적화 | 중간 | 프로파일링 데이터 해석 필요 |
| 레거시 코드 리팩토링 | 낮음 | 히스토리/의도 파악 어려움 |
| UX 의사결정 | 없음 | 사용자 이해 필요 |
| 팀 협업/코드 리뷰 | 낮음 | 맥락+관계 이해 필요 |

## 팩트 4: GitHub DevEx 설문 — 생산성만이 아니다

GitHub이 Wakefield Research와 함께 500명의 미국 기업 개발자를 대상으로 한 설문에서, 흥미로운 관점이 나왔습니다.

> "개발자에게 가장 중요한 건 스토리 포인트나 배포 속도가 아니라, **개발자 경험(DevEx)**이다."

GitHub이 정의한 DevEx 공식:

```
DevEx = (생산성 + 만족감 + 임팩트) × 협업
```

AI 도구의 진짜 가치는 **코드를 빨리 짜는 것**이 아니라:

- **반복 작업에서 정신적 에너지를 아껴서** (87%가 동의)
- **플로우 상태를 유지하게 해줘서** (73%가 동의)
- **더 만족스러운 작업에 집중할 수 있어서** (60~75%가 동의)

한 시니어 엔지니어의 말:

> "Copilot이 있으면 덜 생각해도 돼요. 그리고 생각할 때는 재미있는 걸 생각하게 됩니다."

> 📎 출처: [GitHub — Survey reveals AI's impact on the developer experience](https://github.blog/news-insights/research/survey-reveals-ais-impact-on-the-developer-experience/)

## 팩트 5: 원티드 공고에서 보이는 현실

"AI 쓰면 빨라진다"가 채용 시장에도 반영되고 있을까요? 원티드 프론트엔드 공고 101개 데이터에서:

### "생산성"을 직접 언급한 공고들

> **넥스트그라운드** — 자격요건
> "AI를 적극적으로 활용하여 **생산성을 높인** 경험이 있으신 분"

> **도이치모터스** — 우대사항
> "AI 기반의 코딩 도구를 적극 활용하여 **UI 컴포넌트 개발속도를 높이고** 최적화된 코드를 작성할 수 있는 분"

> **그루우** — 자격요건
> "AI를 활용해 **기존 개발 속도를 압도적으로 뛰어넘은** 경험이 있으신 분"

회사들도 "AI = 생산성"이라는 공식을 기대하고 있어요. 하지만 여기서 중요한 건, **"AI를 써봤다"가 아니라 "AI로 생산성을 높인 경험"**을 요구한다는 겁니다.

---

## 그래서 "10배 빨라진다"는 맞는 말인가요?

### 상황별 정리

| 상황 | 실제 속도 향상 | 비고 |
|------|--------------|------|
| **간단한 보일러플레이트** | 5~10배 | CRUD, 폼 생성 등 |
| **일반 코드 작성** | 1.5~2배 | GitHub 실험 기준 55% |
| **전체 개발 사이클** | 1.2~1.5배 | 코드 작성은 전체의 30~40% |
| **복잡한 문제 해결** | 1~1.2배 | 거의 차이 없음 |
| **처음 접하는 기술** | 2~3배 | 학습 곡선 단축 효과 |
| **레거시 리팩토링** | 0.8~1배 | 오히려 느려질 수 있음 |

**"10배"는 마케팅이고, "1.5~2배"가 현실입니다.** 그래도 1.5배면 엄청난 거예요. 매일 8시간 일하는데, 그 중 2시간을 아낄 수 있다는 뜻이니까요.

## 신입이 알아야 할 것

### 1. AI 도구가 빨라지는 건 맞지만, "뭘" 빨라지게 하는지 이해하세요

보일러플레이트, 반복 코드, 문서화 → AI가 확실히 빠름
설계, 디버깅, 최적화 → 여전히 본인 실력

### 2. "AI 써서 빨라졌다" 경험을 만들어두세요

공고에서 요구하기 시작했어요. 사이드 프로젝트에서 AI 도구 활용한 경험이 이력서에 들어가야 합니다.

> "Cursor를 활용해 컴포넌트 초기 코드 생성 시간을 단축하고, 코드 리뷰와 테스트에 더 많은 시간을 투자했습니다"

이런 한 줄이면 됩니다.

### 3. "빠르게 짜는 것"보다 "잘 판단하는 것"이 중요해집니다

AI가 코드를 빨리 만들어주면, 경쟁력은 **"얼마나 빨리 짜느냐"**에서 **"얼마나 좋은 코드인지 판별하느냐"**로 이동합니다.

기초를 대충 넘기지 마세요. AI가 만든 코드를 읽고 "이건 별로다"라고 말할 수 있어야 합니다.

## 정리: 숫자로 보는 AI 코딩 생산성

| 출처 | 주요 수치 | 의미 |
|------|-----------|------|
| GitHub 실험 (95명) | 작업 완료 **55% 빠름** | 코드 작성 속도 기준 |
| Stack Overflow (전 세계) | **76%**가 AI 사용 중/예정 | 안 쓰면 소수파 |
| Stack Overflow | **45%**가 "복잡한 작업은 못함" | 만능이 아님 |
| Stack Overflow | 호의도 **77% → 72%** 하락 | 기대 vs 현실 괴리 |
| GitHub DevEx (500명) | **87%** 반복작업 정신 에너지 절약 | 속도보다 만족감 |
| 원티드 공고 | **15개** 공고가 AI 생산성 경험 요구 | 채용 시장도 변화 중 |

**AI로 코딩하면 빨라집니다. 하지만 "10배"가 아니라 "1.5배". 그리고 빨라지는 건 "코드 작성"이지 "좋은 소프트웨어 만들기"가 아닙니다.**

진짜 생산성은 빠르게 짜는 게 아니라, 중요한 것에 시간을 쓰는 것입니다. AI가 반복 작업을 대신해주면, 그 시간에 설계하고, 테스트하고, 더 좋은 사용자 경험을 고민하세요.
