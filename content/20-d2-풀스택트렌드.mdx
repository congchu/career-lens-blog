## 들어가며

프론트엔드 공고인데 자격요건을 보면:

> "AWS 경험"
> "Docker 사용 경험"
> "Node.js 백엔드 개발 가능하신 분"
> "CI/CD 파이프라인 구축 경험"

이건 프론트엔드 공고가 맞나요? 아니면 풀스택인가요?

**맞습니다. 프론트엔드 공고입니다.** 하지만 2026년의 프론트엔드는 예전과 다릅니다.

---

## 숫자로 보는 "프론트엔드의 확장"

프론트엔드 공고 101개에서 **백엔드/인프라 관련 키워드**를 추출했습니다.

| 키워드 | 등장 수 | 비율 |
|--------|---------|------|
| **Node.js** | 10개 | 10% |
| **AWS** | 9개 | 9% |
| **REST API** | 8개 | 8% |
| **CI/CD** | 6개 | 6% |
| **Docker** | 4개 | 4% |
| **GraphQL** | 3개 | 3% |
| **monorepo/Turborepo** | 6개 | 6% |

**10개 중 1개 공고가 Node.js를 요구하고, 9개가 AWS를 요구합니다.** 프론트엔드라고 해서 브라우저 안에만 머물 수 없는 시대가 됐어요.

---

## 왜 이렇게 됐나?

### 이유 1: Next.js가 서버 영역을 먹었다

Next.js를 쓰면 프론트엔드 개발자가 자연스럽게 서버 코드를 작성하게 됩니다.

- **Server Component** — 서버에서 렌더링되는 React 컴포넌트
- **API Routes / Route Handlers** — 간단한 백엔드 API를 Next.js 안에서 직접 작성
- **Server Actions** — 폼 제출을 서버에서 직접 처리

공고 28개(28%)가 Next.js를 요구하는데, Next.js를 쓰면 **배포(AWS/Vercel), 서버 사이드 로직, API 설계**를 어느 정도 알아야 합니다.

### 이유 2: 스타트업에서는 경계가 없다

신입을 뽑는 12개 공고 중 80%가 50인 이하 스타트업이었죠. 소규모 팀에서는:

- 프론트 따로, 백엔드 따로 뽑을 여유가 없음
- "프론트엔드 중심 + 간단한 API는 직접" 기대
- 배포/인프라도 개발자가 직접 하는 경우 많음

### 이유 3: BFF(Backend For Frontend) 패턴

마이크로서비스 아키텍처에서, 프론트엔드 팀이 **자신만의 API 서버(BFF)**를 운영하는 패턴이 늘고 있습니다. 이러면 프론트엔드 개발자가 Node.js로 BFF를 관리하게 돼요.

---

## 그렇다고 풀스택이 되라는 건 아닙니다

여기서 오해하면 안 되는 게:

| 공고가 원하는 것 | 공고가 원하지 않는 것 |
|----------------|---------------------|
| AWS에 배포할 수 있는 정도 | AWS 아키텍처 설계 |
| 간단한 API를 만들 수 있는 수준 | 복잡한 백엔드 시스템 구축 |
| Docker 이미지를 빌드/실행 | Kubernetes 클러스터 운영 |
| CI/CD를 설정할 수 있는 정도 | DevOps 전담 |

**프론트엔드 "중심"이되, 주변 영역을 이해하는 T자형 인재**를 원하는 겁니다.

```
깊이
│
│  ★ React/TypeScript/Next.js (깊이)
│  ★
│  ★
│  ★
│  ─────────────────────── (넓이)
│  Node  AWS  Docker  CI/CD
│
└────────────────────────── 넓이
```

## 신입이 당장 해볼 수 있는 것

### 배포 (가장 쉬움)

1. **Vercel**에 Next.js 프로젝트 배포 — 10분이면 됨
2. **GitHub Actions**로 자동 배포 설정 — 30분
3. 이력서에 "CI/CD 파이프라인 구축 경험" 한 줄 추가

이것만으로 **6%의 공고에서 요구하는 CI/CD 경험**을 만족할 수 있어요.

### API 만들기 (중간)

1. **Next.js Route Handlers**로 간단한 API 만들기
2. 또는 Express.js로 REST API 하나 만들어보기
3. 프론트엔드 프로젝트에 "직접 만든 API" 연결

> "Next.js API Routes로 사용자 인증과 데이터 CRUD API를 구현했습니다"

이 한 줄이 이력서에 있으면, **Node.js 경험 + API 설계 경험**이 됩니다.

### Docker (선택)

```bash
# 이거 한 번 해보는 것만으로도 경험
docker build -t my-app .
docker run -p 3000:3000 my-app
```

Dockerfile 만들고 이미지 빌드하는 과정을 한 번이라도 경험해보면, 면접에서 "Docker 써봤냐"는 질문에 "네"라고 할 수 있어요.

### AWS (선택)

- **S3 + CloudFront**로 정적 사이트 배포
- 또는 **EC2** 하나 띄워서 Node.js 서버 실행

무료 티어로 충분하고, "AWS에 직접 배포해본 경험"이 생깁니다.

## "풀스택" 공고 vs "프론트엔드 + 알파" 공고 구분법

| | 프론트 + 알파 (괜찮음) | 진짜 풀스택 (주의) |
|---|---|---|
| 공고 제목 | "프론트엔드 개발자" | "풀스택 개발자" |
| 기술 비율 | 프론트 70% + 백엔드 30% | 프론트 50% + 백엔드 50% |
| 팀 구성 | 백엔드 개발자가 별도로 있음 | 개발자 1~2명 |
| 업무 설명 | "프론트 중심, 필요시 API" | "프론트부터 인프라까지 전부" |

**"프론트엔드 + 알파"는 성장에 좋습니다.** 하지만 **"사실상 풀스택인데 프론트엔드라고 부르는"** 공고는 번아웃 위험이 있어요.

## 정리: 2026년 프론트엔드의 범위

```
[필수 영역]
React + TypeScript + Next.js + CSS

[확장 영역 — 알면 유리]
Node.js(API) + AWS(배포) + Docker + CI/CD

[백엔드 영역 — 프론트엔드에게 기대하지 않는 것]
데이터베이스 설계, 마이크로서비스, 쿠버네티스
```

**프론트엔드 개발자에게 AWS를 요구하는 건, "인프라 전문가가 되라"는 게 아닙니다.** "내가 만든 걸 세상에 내보낼 수 있는 능력"을 원하는 거예요.

Vercel에 배포할 수 있으면 충분하고, 거기서 한 걸음 더 나아가면 차별화가 됩니다.
