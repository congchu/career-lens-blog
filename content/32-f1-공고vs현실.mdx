## 들어가며

채용공고에는 React, TypeScript, Next.js가 적혀있었습니다.
입사 첫날, 열어본 코드베이스에는 jQuery와 JSP가 있었습니다.

**실화입니다.** 개발자 커뮤니티에서 꾸준히 나오는 이야기예요.

공고에 써있는 기술 스택이 실제 업무와 다른 경우가 왜 생기는 걸까요? 101개 공고 데이터를 분석해서, **어떤 공고가 "진짜"이고 어떤 공고가 "기대"인지** 구분하는 법을 알려드릴게요.

---

## 공고에 숨어있는 단서들

### 101개 중 12개(12%)가 "레거시"를 직접 언급합니다

| 키워드 | 등장 수 | 의미 |
|--------|---------|------|
| **유지보수** | 31개 (31%) | 기존 코드를 관리하는 업무 포함 |
| **레거시/Legacy** | 14개 (14%) | 오래된 코드베이스 존재 |
| **리팩토링** | 12개 (12%) | 기존 코드를 개선하는 업무 |
| **마이그레이션** | 4개 (4%) | 기술 전환 진행 중 |

**"유지보수" 31개.** 공고 3개 중 1개는 새로운 걸 만드는 게 아니라, **이미 있는 걸 고치는 업무**를 포함하고 있다는 뜻이에요.

---

## 공고 vs 현실, 괴리가 생기는 3가지 패턴

### 패턴 1: "전환 중" — 레거시에서 React로 이동하는 회사

> 레디포스트: "레거시의 안전한 해체와 재구성이라는 명확한 기술적 전환 국면에 서 있습니다"
> 랭디: "서비스를 Next.js + React로 마이그레이션"
> 커넥트웨이브: "레거시 → 신규 플랫폼 전환 기술 리딩"

이런 공고는 **솔직한 편**입니다. "우리 레거시 있고, React로 바꾸는 중"이라고 말해주니까요.

**함정**: React를 쓰긴 쓰는데, 업무의 50%는 레거시 코드 분석과 마이그레이션입니다. **새로운 기능 개발만 기대하고 입사하면 괴리를 느낄 수 있어요.**

**판별법**: "마이그레이션", "전환", "리팩토링"이 주요 업무에 있으면 → 레거시 비중 높음

### 패턴 2: "희망 사항" — 쓰고 싶은 기술을 공고에 적은 회사

공고에 React, TypeScript, Next.js를 써놨지만, 실제로는:
- 아직 도입 전이거나
- 일부만 적용되어 있거나
- "앞으로 쓸 예정"인 경우

이건 데이터에서 직접 보이지 않지만, **원티드 skill_tags에 React가 있는데 본문에 레거시를 언급하는 공고가 8개**있었습니다.

예를 들어 한 회사는 skill_tags에 React가 있지만, 자격요건에 "다양한 레거시 소스 코드 분석 경험(C#, Spring Framework)"을 요구해요.

**판별법**: 기술 태그와 자격요건 본문이 다르면 → 태그는 "희망", 본문이 "현실"

### 패턴 3: "부분 적용" — 신규는 React, 기존은 레거시

가장 흔한 패턴입니다. 대부분의 회사가 이 상태예요.

> "신규 서비스는 React + TypeScript로 개발합니다"
> "기존 서비스 유지보수도 담당합니다"

이 두 문장이 같은 공고에 있으면, **React도 쓰고 레거시도 만진다**는 뜻입니다. 공고가 틀린 건 아닌데, React "만" 하는 건 아닌 거죠.

**판별법**: "유지보수" + "신규 개발"이 동시에 있으면 → 반반

## 괴리를 줄이는 5가지 체크 방법

### 1. 기술 블로그 확인

회사 기술 블로그가 있으면 **실제로 어떤 기술을 쓰는지** 알 수 있어요.

- 블로그에 React 관련 글이 있다 → 진짜 씀
- 블로그가 2년 전에 멈췄다 → 기술 투자에 관심 적을 수 있음
- 블로그 자체가 없다 → 판단 불가 (나쁜 건 아님)

### 2. "기술 스택" 섹션의 구체성

| 구체적 (신뢰) | 모호 (주의) |
|-------------|-----------|
| "React 18, Next.js 14 App Router" | "React" |
| "TypeScript strict mode" | "JavaScript/TypeScript" |
| "Zustand + React Query" | "상태관리 도구" |
| "Vitest + Testing Library" | "테스트" |

**버전까지 써있으면 실제로 쓰고 있을 확률이 높습니다.** "React"만 덜렁 써있으면 아직 도입 전일 수도 있어요.

### 3. "주요 업무" vs "자격 요건" 비교

- **주요 업무**에 "유지보수, 레거시 개선"이 있다 → 현실
- **자격 요건**에만 "React, TypeScript" → 기대

**주요 업무 = 실제로 할 일**, 자격 요건 = 이 일을 하려면 필요한 기술. 주요 업무를 먼저 읽으세요.

### 4. 면접에서 직접 물어보기

면접관에게 이렇게 물어보세요:

> "현재 프론트엔드 코드베이스에서 React와 레거시 코드의 비율이 어떻게 되나요?"
> "신규 기능은 모두 React로 개발하나요?"
> "레거시 코드 유지보수 업무가 어느 정도 비중인가요?"

이걸 물어봤는데 **답변이 모호하면**, 레거시 비중이 높을 확률이 큽니다.

### 5. 잡플래닛/블라인드 후기

현직자 후기에서 "기술 스택" 관련 멘트를 찾아보세요. "공고랑 다르다"는 후기가 있으면 경고 신호.

## 레거시가 무조건 나쁜 건 아닙니다

여기서 오해하면 안 되는 게 있어요.

### 레거시 = 나쁜 회사?

**아닙니다.** 레거시가 있다는 건 **"서비스가 오래 살아남았다"**는 뜻이기도 해요.

- 카카오도 레거시 있습니다
- 네이버도 레거시 있습니다
- 토스도 초기 코드는 레거시입니다

### 레거시 업무에서 배울 수 있는 것

1. **코드 읽기 능력** — 남이 쓴 코드를 분석하는 건 실무 핵심 역량
2. **점진적 개선** — 전부 갈아엎는 게 아니라 조금씩 개선하는 기술
3. **마이그레이션 경험** — "jQuery → React 전환했습니다"는 이력서에 강력한 한 줄
4. **비즈니스 이해** — 레거시 코드를 이해하려면 비즈니스 로직을 깊이 파악해야 함

**문제는 레거시 자체가 아니라, "공고와 현실이 다를 때의 실망감"입니다.**

## 정리: 공고의 기술 스택을 읽는 법

| 공고 표현 | 현실 해석 |
|-----------|----------|
| "React, TypeScript" (기술 스택에만) | 쓰고 있거나, 쓸 예정 |
| "React 18, Next.js 14 App Router" | 실제로 쓰고 있음 (구체적) |
| "유지보수 및 신규 개발" | React + 레거시 반반 |
| "레거시 → React 마이그레이션" | 레거시가 주, React 전환 중 |
| "리팩토링 및 코드 품질 개선" | 기존 코드 개선이 주 업무 |

**공고는 회사의 "이상"이고, 주요 업무는 "현실"입니다.** 기술 스택 태그보다 주요 업무 설명을 꼼꼼히 읽으세요.

그리고 면접에서 꼭 물어보세요. **"실제로 React를 얼마나 쓰고 있나요?"** 이 질문 하나가 입사 후 괴리를 90% 줄여줍니다.
