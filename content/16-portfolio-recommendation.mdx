## 들어가며

포트폴리오 뭘 만들지 고민되시죠?

그래서 대부분 이런 걸 만듭니다:
- 투두앱
- 날씨앱
- 영화 검색앱
- 개인 블로그

솔직히 말할게요. **면접관은 이걸 몇백 번 봤습니다.** 투두앱이 나쁜 건 아니지만, **차별화가 안 됩니다.**

그러면 뭘 만들어야 할까? 감으로 정하지 말고, **회사들이 실제로 뭘 만드는지** 데이터를 보고 정합시다.

## 공고 101개에서 추출한 프로젝트 도메인

원티드 프론트엔드 공고 101개의 업무 설명을 분석해서, **실제로 어떤 유형의 프로젝트를 만드는지** 카운팅했습니다.

| 프로젝트 도메인 | 공고 수 | 비율 |
|----------------|---------|------|
| **SaaS/B2B 플랫폼** | 81개 | 80% |
| **모바일/웹앱** | 63개 | 62% |
| **대시보드/어드민** | 39개 | 39% |
| **이커머스/쇼핑** | 23개 | 23% |
| **실시간 기능 (채팅/알림)** | 22개 | 22% |
| **데이터 시각화** | 13개 | 13% |
| **접근성 (a11y)** | 11개 | 11% |
| **디자인 시스템** | 10개 | 10% |

투두앱은 이 리스트에 **없습니다.**

## 포트폴리오 추천 TOP 5

공고 데이터를 기반으로, **면접에서 어필되는 포트폴리오** 5가지를 추천합니다.

### 1. 대시보드/어드민 (39% 공고가 관련)

**만들 것**: 데이터를 보여주는 관리자 페이지

**왜 강력한가**:
- 전체 공고의 39%가 대시보드/어드민 개발을 업무에 포함
- 테이블, 필터, 페이지네이션, 검색 — **실무에서 매일 쓰는 패턴**
- CRUD 전체 사이클을 보여줄 수 있음

**구체적 예시**:
- 채용공고 분석 대시보드 (공고 목록 + 필터 + 통계 차트)
- 쇼핑몰 주문 관리 페이지
- 프로젝트 관리 도구 (칸반 보드)

**넣어야 할 기술**: React + TypeScript + React Query(서버 상태) + 테이블 라이브러리

### 2. 실시간 기능이 있는 앱 (22% 공고가 관련)

**만들 것**: 채팅, 실시간 알림, 협업 기능

**왜 강력한가**:
- WebSocket/실시간 데이터는 **투두앱과 확실히 차별화**
- 상태관리가 복잡해지는 경험 → 면접에서 이야기할 거리가 풍부
- 22%의 공고가 실시간 기능을 언급

**구체적 예시**:
- 실시간 채팅방 (WebSocket + 메시지 히스토리)
- 실시간 협업 메모장 (동시 편집)
- 주식/코인 실시간 가격 대시보드

**넣어야 할 기술**: WebSocket/SSE + 상태관리(Zustand) + 낙관적 업데이트

### 3. 이커머스 기능 (23% 공고가 관련)

**만들 것**: 상품 목록 + 장바구니 + 결제 플로우

**왜 강력한가**:
- 이커머스는 프론트엔드의 **가장 흔한 도메인**
- 복잡한 상태 관리 (장바구니, 필터, 정렬)
- 성능 최적화 포인트가 많음 (이미지 최적화, 무한 스크롤, SSR)

**구체적 예시**:
- 쇼핑몰 (상품 리스트 + 검색/필터 + 장바구니 + 결제 UI)
- 중고거래 플랫폼 (이미지 업로드 + 채팅)
- 구독 서비스 랜딩 + 가격표 페이지

**넣어야 할 기술**: Next.js(SSR/ISR) + 이미지 최적화 + React Query + 무한 스크롤

### 4. 디자인 시스템 / 컴포넌트 라이브러리 (10%)

**만들 것**: 재사용 가능한 UI 컴포넌트 세트

**왜 강력한가**:
- "컴포넌트 설계 능력"을 직접적으로 보여줌
- Storybook 경험은 **우대사항 단골**
- 10%의 공고가 디자인 시스템 업무를 포함

**구체적 예시**:
- Button, Input, Modal, Toast 등 기본 컴포넌트 세트
- Storybook으로 문서화
- npm 패키지로 배포까지

**넣어야 할 기술**: Storybook + TypeScript(제네릭) + 접근성(a11y) + npm 배포

### 5. 데이터 시각화 (13%)

**만들 것**: 차트와 그래프로 데이터를 보여주는 페이지

**왜 강력한가**:
- 대시보드와 결합하면 시너지
- SVG, Canvas 등 고급 렌더링 기술
- 13%의 공고가 데이터 시각화를 언급

**구체적 예시**:
- GitHub 활동 시각화 (잔디 + 기여도 차트)
- 개인 지출 분석 대시보드
- 운동 기록 트래커 (차트 + 통계)

**넣어야 할 기술**: Chart.js/Recharts/D3 + 반응형 + 다크모드

## 투두앱이 나쁜 게 아니라, "투두앱만"이 문제입니다

투두앱으로 **배운 것**은 가치 있어요:
- React 상태관리 기초
- CRUD 패턴
- 이벤트 핸들링

하지만 포트폴리오에 넣을 때는, **투두앱에서 배운 걸 더 복잡한 프로젝트에 적용**했다는 걸 보여줘야 합니다.

| 투두앱에서 배운 것 | 포트폴리오에서 보여줄 것 |
|-------------------|----------------------|
| useState로 상태관리 | Zustand + React Query로 복잡한 상태 |
| 로컬 CRUD | API 연동 + 에러/로딩 처리 |
| 단순 리스트 렌더링 | 무한 스크롤 + 가상화 |
| CSS 스타일링 | 반응형 + 다크모드 + 접근성 |

## 포트폴리오 차별화 체크리스트

포트폴리오를 만든 후 아래를 확인하세요:

### 기술적 차별화
- [ ] **TypeScript** 사용 (공고 40%가 요구)
- [ ] **React Query/TanStack** 서버 상태 관리 (트렌드)
- [ ] **테스트 코드** 1개 이상 (우대사항 단골)
- [ ] **CI/CD** 자동 배포 (GitHub Actions)
- [ ] **반응형** 모바일 대응

### 프로젝트 품질
- [ ] **README**가 상세한가? (기술 선택 이유, 아키텍처 설명)
- [ ] **배포**되어 있는가? (Vercel/Netlify 등)
- [ ] **성능 최적화** 경험이 있는가? (Lighthouse 점수)
- [ ] **에러 처리**가 되어 있는가? (빈 상태, 네트워크 에러)

### 이야기 차별화
- [ ] "왜 이 프로젝트를 만들었나?" — **동기**
- [ ] "가장 어려웠던 점은?" — **문제 해결**
- [ ] "다시 만든다면 뭘 바꿀까?" — **성장**

이 3가지 질문에 답할 수 있으면, 면접에서 할 이야기가 풍부해집니다.

## 현실적 제안: 1개를 제대로 만드세요

5개를 대충 만드는 것보다, **1개를 깊이 있게 만드는 게 훨씬 낫습니다.**

추천 조합:

> **대시보드 + 실시간 기능** = "실시간 데이터 대시보드"
> React + TypeScript + React Query + WebSocket + 차트

이거 하나면:
- 대시보드 (39%)
- 실시간 (22%)
- 데이터 시각화 (13%)
- 3개 도메인을 커버합니다

그리고 이 프로젝트를 면접에서 이렇게 말할 수 있어요:

> "WebSocket으로 실시간 데이터를 받아서 차트를 업데이트하는 대시보드를 만들었는데, 데이터가 많아지면 리렌더링이 느려져서 가상화와 useMemo를 적용해 성능을 개선했습니다."

투두앱으로는 절대 나올 수 없는 이야기죠.
