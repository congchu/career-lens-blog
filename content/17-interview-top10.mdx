## 들어가며

면접 준비할 때 가장 막막한 게, **"뭘 물어볼지 모른다"**는 거예요.

그래서 구글에 "프론트엔드 면접 질문"을 검색하면 200개짜리 리스트가 나오는데... 다 외울 수도 없고, 실제로 물어보는 것만 준비하고 싶잖아요.

방법이 있습니다. **채용공고의 자격요건을 보면 면접 질문이 보여요.**

공고에 "React 경험"이라고 써있으면 → React 관련 질문이 나옵니다.
공고에 "TypeScript"가 있으면 → 타입 관련 질문이 나옵니다.

단순하죠? 이걸 101개 공고 데이터로 **빈도순**으로 정리했습니다.

## 빈출 질문 TOP 10 — 답변 포인트 포함

### Q1. "React에서 상태 관리를 어떻게 하시나요?"

**출제 근거**: React 65개 + 상태관리 20개+ = 거의 모든 면접에서 나옴

**기본 답변**: useState, useReducer로 로컬 상태 관리

**레벨업 답변**:
- "로컬 상태는 useState, 글로벌 상태는 Zustand, 서버 상태는 React Query로 분리합니다"
- **왜 레벨업인가**: 공고 8개가 Zustand를, 7개가 TanStack/React Query를 언급. 상태 분류(로컬/글로벌/서버)를 이해하고 있다는 걸 보여주면 차별화.

### Q2. "TypeScript를 왜 사용하나요? 장단점은?"

**출제 근거**: TypeScript 40개 (40%)

**기본 답변**: 타입 안전성, 자동완성, 버그 방지

**레벨업 답변**:
- "런타임 에러를 컴파일 타임에 잡을 수 있어요. 특히 API 응답 타입을 정의하면 undefined 에러를 방지할 수 있습니다"
- "제네릭을 활용하면 재사용 가능한 컴포넌트를 타입 안전하게 만들 수 있어요"
- 단점도 말할 것: "초기 설정 비용, 타입 정의에 시간 소요, 과도한 타입 추상화의 위험"

### Q3. "Virtual DOM이 뭔가요? 왜 쓰나요?"

**출제 근거**: React 65개 — React 면접의 클래식

**핵심**: "실제 DOM 조작은 비싸서, 가상 DOM에서 변경점(diff)을 계산한 후 최소한의 실제 DOM 업데이트만 수행합니다"

**레벨업**: "React 18에서는 Concurrent Mode와 Automatic Batching으로 더 효율적으로 동작합니다"

### Q4. "useEffect의 동작 원리를 설명해주세요"

**출제 근거**: React 기초 — 사실상 필수

**핵심**:
- 컴포넌트 렌더링 후 실행 (비동기)
- 의존성 배열로 실행 조건 제어
- 클린업 함수로 메모리 누수 방지

**레벨업**: "useEffect 안에서 API를 호출하면 race condition이 생길 수 있어서, AbortController를 사용하거나 React Query를 쓰는 게 더 안전합니다"

### Q5. "SSR과 CSR의 차이점은?"

**출제 근거**: Next.js 28개 (28%)

**기본 답변**: SSR은 서버에서 HTML 생성, CSR은 브라우저에서 JavaScript로 렌더링

**레벨업 답변**:
- "Next.js App Router에서는 Server Component가 기본이고, 클라이언트 인터랙션이 필요한 부분만 'use client'로 분리합니다"
- "SSG, ISR도 설명할 수 있으면 플러스" — "블로그 같은 정적 콘텐츠는 SSG, 상품 페이지처럼 자주 바뀌는 건 ISR"

### Q6. "브라우저에 URL을 입력하면 무슨 일이 일어나나요?"

**출제 근거**: JavaScript/웹 기초 — 프론트엔드 면접 단골

**답변 포인트**: DNS 조회 → TCP 연결 → HTTP 요청 → 서버 응답 → HTML 파싱 → CSSOM + DOM → 렌더 트리 → 레이아웃 → 페인트

**레벨업**: "CRP(Critical Rendering Path) 최적화 — CSS는 head에, JS는 defer로, 이미지는 lazy loading"

### Q7. "클로저(Closure)가 뭔가요?"

**출제 근거**: JavaScript 24개 — JS 기초 필수

**핵심**: "함수가 선언될 때의 렉시컬 환경을 기억하는 것"

**실전 예시**: "React의 useState가 클로저를 활용합니다. 이전 상태를 기억하고 있다가 setState가 호출될 때 참조하죠"

### Q8. "웹 성능 최적화 경험이 있나요?"

**출제 근거**: CSS 29개 + 실무 역량 평가

**답변 패턴**: 측정 → 문제 발견 → 개선 → 결과

**구체적 예시**:
- "Lighthouse로 측정했더니 LCP가 4초 → 이미지 최적화(WebP 변환, lazy loading)와 코드 스플리팅으로 1.8초로 개선"
- "불필요한 리렌더링을 React DevTools Profiler로 찾아서 useMemo/memo 적용"

### Q9. "Git 충돌 해결 경험이 있나요?"

**출제 근거**: Git 28개 (28%)

**핵심**: 단순히 "해결했습니다"가 아니라 **과정을 설명**

**답변**: "feature 브랜치를 main에 머지할 때 충돌이 발생했고, git diff로 변경 내역을 확인한 후 팀원과 의논해서 어떤 코드를 유지할지 결정했습니다"

### Q10. "테스트 코드를 작성해본 적 있나요?"

**출제 근거**: Jest/Vitest 등 우대사항 + 코드 품질 평가

**없는 경우**: "아직 경험은 없지만, 최근 사이드 프로젝트에 Vitest + Testing Library를 도입해보고 있습니다"

**있는 경우**: "핵심 비즈니스 로직과 사용자 인터랙션 위주로 테스트를 작성했고, CI에서 자동 실행되도록 설정했습니다"

## 면접관이 진짜 보는 것

질문의 "정답"보다 중요한 건:

1. **"모르겠습니다"를 솔직히 말하는 것** — 모르는 걸 아는 척하면 바로 티남
2. **생각하는 과정을 보여주는 것** — "정확히는 모르지만, 이런 원리일 것 같습니다" 
3. **경험과 연결하는 것** — "이론적으로는 이렇고, 제가 프로젝트에서 경험한 바로는..."
4. **한계를 인정하고 학습 의지를 보이는 것** — "아직 깊이 있게 다뤄보지 못했는데, 이런 방향으로 공부하고 있습니다"

신입 면접에서 **모든 걸 알기를 기대하지 않습니다.** 기본기가 탄탄하고, 배울 자세가 있는지를 봐요.
