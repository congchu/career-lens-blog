## 들어가며

알고리즘 문제 풀기 싫어서, "코딩테스트 안 보는 회사"를 찾는 분들 많으시죠.

그래서 코테 없는 회사에 지원했는데... **사전과제로 2주짜리 프로젝트가 나왔습니다.**

코테를 안 본다고 쉬운 게 아닙니다. **대신 다른 방식으로 검증합니다.** 뭘 준비해야 하는지 알아야 효율적으로 대비할 수 있어요.

---

## 프론트엔드 채용 프로세스 4가지 유형

### 유형 1: 코딩테스트형

> 서류 → **코딩테스트** → 기술면접 → 최종면접

**주로 쓰는 곳**: 대기업, 중견기업, 네카라쿠배 계열
**특징**: 알고리즘/자료구조 문제 2~4개, 시간제한 1~3시간
**준비법**: 프로그래머스, 백준 (BFS/DFS, DP, 해시, 정렬 위주)

### 유형 2: 사전과제형

> 서류 → **사전과제** → 기술면접 (과제 리뷰) → 최종면접

**주로 쓰는 곳**: 스타트업, 중소기업
**특징**: 실제 프론트엔드 과제 (API 연동 앱, 대시보드, 특정 기능 구현)
**기간**: 3일~2주

공고 101개 중 **7개(7%)**가 "과제"를 직접 언급했습니다.

**사전과제에서 보는 것**:
- 코드 구조와 컨벤션
- 컴포넌트 설계 능력
- 상태관리 패턴
- 에러 처리
- Git 커밋 히스토리 (!)

### 유형 3: 포트폴리오형

> 서류 + **포트폴리오** → 기술면접 (포폴 리뷰) → 최종면접

**주로 쓰는 곳**: 디자인 에이전시, 프로덕트 중심 회사
**특징**: 이전 프로젝트를 보고 판단

공고 101개 중 **8개(8%)**가 "포트폴리오"를 요구했습니다.

### 유형 4: 면접중심형

> 서류 → **기술면접** (코드 없이 대화) → 최종면접

**주로 쓰는 곳**: 경력직 채용, 시니어급
**특징**: 화이트보드 설계, 기술 토론, 경험 기반 질문
**신입에게는**: 드물지만, 소규모 스타트업에서 가끔 있음

## 유형별 난이도 비교

| | 코딩테스트형 | 사전과제형 | 포트폴리오형 | 면접중심형 |
|---|---|---|---|---|
| 준비 시간 | 수개월 | 3일~2주 | 수주~수개월 | 수주 |
| 평가 기준 | 알고리즘 실력 | 실무 코딩 능력 | 프로젝트 경험 | 기술 이해도 |
| 공정성 | 높음 | 중간 | 낮음 | 낮음 |
| 신입 유리도 | 중간 | 높음 | 중간 | 낮음 |
| 스트레스 | 시험 당일 집중 | 과제 기간 내내 | 만드는 데 오래 | 면접 자체 |

**"코테 안 보는 회사 = 쉽다"가 아닌 이유:**

사전과제형은 실제로 **코드를 짜서 제출**해야 합니다. 알고리즘은 정답이 있지만, 사전과제는 **"좋은 코드"의 기준이 회사마다 다릅니다.** 어떤 회사는 완성도를 보고, 어떤 회사는 코드 구조를 보고, 어떤 회사는 Git 히스토리를 봐요.

---

## 사전과제 잘 하는 법 (가장 실용적인 파트)

코테 대신 사전과제를 내는 회사가 늘고 있으니, 이걸 잘 하는 게 중요합니다.

### 1. 완성도보다 구조를 보여주세요

100% 기능 구현 < 80% 구현 + 깔끔한 코드

면접관은 **"이 사람이 팀에서 일하면 어떤 코드를 쓸까"**를 봅니다.

```
// 나쁜 예: 한 파일에 500줄
src/App.tsx (모든 로직)

// 좋은 예: 역할별 분리
src/
├── components/    (UI 컴포넌트)
├── hooks/         (커스텀 훅)
├── api/           (API 호출)
├── types/         (타입 정의)
└── utils/         (유틸리티)
```

### 2. Git 커밋을 의미 있게 나누세요

면접관이 Git 히스토리를 봅니다. 진짜로요.

```
// 나쁜 예
"init"
"update"
"fix"
"done"

// 좋은 예
"feat: 상품 목록 API 연동 및 로딩 상태 처리"
"refactor: 상품 카드 컴포넌트 분리"
"fix: 무한 스크롤 마지막 페이지 중복 요청 방지"
"style: 반응형 레이아웃 적용 (768px 브레이크포인트)"
```

### 3. README를 제대로 쓰세요

```markdown
## 기술 스택
React 18 + TypeScript + React Query + Zustand

## 실행 방법
npm install && npm run dev

## 기술 선택 이유
- React Query: 서버 상태 캐싱 + 자동 리페치
- Zustand: 클라이언트 상태 (최소한의 보일러플레이트)

## 고민한 점 / 트레이드오프
- 무한 스크롤 vs 페이지네이션 → UX 기준으로 무한 스크롤 선택
- 에러 바운더리 적용 범위 → 페이지 단위로 결정
```

**"고민한 점"이 가장 중요합니다.** 이걸 쓰면 면접에서 자연스럽게 대화가 이어져요.

### 4. 테스트 코드 하나라도 넣으세요

전부 테스트할 필요 없어요. **핵심 비즈니스 로직 1~2개만 테스트**하면 됩니다.

> "시간이 부족해서 전체 테스트는 못 했지만, 핵심 필터링 로직에 대한 유닛 테스트를 작성했습니다."

이 한마디가 차별화됩니다.

### 5. 배포까지 하세요

Vercel이나 Netlify에 배포하면 **3분**이면 됩니다. 면접관이 직접 확인할 수 있으니까, 로컬에서만 돌아가는 것보다 훨씬 인상적이에요.

## 채용 프로세스를 미리 아는 법

### 방법 1: 공고에서 힌트 찾기

| 공고 문구 | 예상 프로세스 |
|-----------|-------------|
| "코딩테스트" 직접 언급 | 코테형 |
| "사전과제", "기술과제" | 과제형 |
| "포트폴리오 필수" | 포폴형 |
| 프로세스 설명이 없음 | 면접에서 확인 필요 |

### 방법 2: 잡플래닛/블라인드 면접 후기

"면접 후기"에 채용 프로세스가 상세히 나와 있는 경우가 많아요.

### 방법 3: 지원 후 인사팀에 물어보기

서류 합격 후 "채용 프로세스가 어떻게 되나요?"라고 물어보는 건 전혀 이상하지 않습니다. 오히려 준비성 있어 보여요.

---

## 신입에게 유리한 프로세스는?

### 사전과제형이 가장 유리합니다

**이유**:
1. 알고리즘이 약해도 실무 코딩으로 승부 가능
2. 시간을 들여서 최선의 코드를 보여줄 수 있음
3. 포트폴리오가 약해도 과제로 어필 가능
4. 과제 자체가 면접 대화 소재가 됨

**단점**: 시간 투자가 큼. 여러 회사에 동시 지원하면 과제가 겹칠 수 있음.

### 코테형은 양날의 검

알고리즘을 잘하면 서류 스펙과 무관하게 통과 가능. 하지만 프론트엔드 실무와 거리가 있어서, 코테 통과 후 기술면접에서 떨어지는 경우도 많아요.

---

## 정리

| 오해 | 현실 |
|------|------|
| 코테 없으면 쉽다 | 대신 과제/포폴로 검증 (더 시간 소요) |
| 알고리즘만 잘하면 된다 | 프론트엔드는 실무 코딩 능력이 더 중요 |
| 프로세스는 지원 후에 알 수 있다 | 공고 + 후기 + 문의로 미리 파악 가능 |

**코딩테스트를 안 보는 회사는 "쉬운 회사"가 아니라, "다른 방식으로 검증하는 회사"입니다.** 어떤 방식이든 준비가 필요하고, 자신에게 유리한 유형을 알고 전략적으로 지원하는 게 중요해요.
