## 들어가며

"AI가 코드를 다 짜주면, 개발자는 뭘 하나요?"

취준생들이 요즘 가장 많이 하는 질문입니다. 솔직히 무섭기도 하죠. Cursor에 "로그인 페이지 만들어줘" 하면 진짜 만들어주는 시대니까요.

그런데 결론부터 말하면 — **개발자가 할 일은 오히려 늘어나고 있습니다.**

다만, "뭘 하느냐"가 달라졌어요.

---

## AI가 잘하는 것 vs 못하는 것

먼저 현실을 직시해봅시다. AI 코딩 도구가 진짜 잘하는 것과 아직 못하는 것을 나눠보면:

### AI가 잘하는 것

- **반복적인 코드 작성** — CRUD, 폼 컴포넌트, API 연동 보일러플레이트
- **패턴이 명확한 작업** — "이 디자인 보고 CSS 짜줘", "이 API 스펙 보고 타입 만들어줘"
- **코드 변환** — JavaScript → TypeScript, Class → Hooks, REST → GraphQL
- **문서화** — JSDoc 생성, README 작성, 주석 추가
- **버그 찾기** — 에러 메시지 해석, 코드 리뷰 보조

### AI가 못하는 것

- **"왜 이렇게 만들어야 하는지" 판단하기** — 비즈니스 로직의 이유
- **사용자 경험 설계** — "이 버튼을 여기에 두는 게 맞나?"
- **팀과 소통하기** — "디자이너가 이걸 원한 건 이런 의미야"
- **장기적 아키텍처 결정** — "이 프로젝트에 Next.js vs Remix 중 뭐가 맞아?"
- **맥락 이해** — "이 회사의 사용자는 이런 특성이 있어서..."
- **품질 판단** — AI가 만든 코드가 정말 좋은 코드인지 판별

Stack Overflow 2024 설문에서도 이걸 보여주는 데이터가 있습니다. 전문 개발자의 **45%가 "AI 도구는 복잡한 작업을 잘 못한다"**고 답했어요. 반면 코드 작성(82%)이나 문서화에서는 대부분이 AI를 활용하고 있었고요.

> 📎 출처: [Stack Overflow 2024 Developer Survey — AI](https://survey.stackoverflow.co/2024/ai)

쉽게 말하면, **AI는 "어떻게(How)" 코드를 짜는지는 잘 알지만, "왜(Why)" 이렇게 짜야 하는지는 모릅니다.**

---

## 그래서 개발자의 역할은 어떻게 바뀌나?

### Before AI: 코드를 "직접" 짜는 사람

```
기획서 읽기 → 설계 → 코드 작성 → 테스트 → 배포
              ↑ 여기에 시간의 70%
```

### After AI: 코드를 "지시하고 검증"하는 사람

```
기획서 읽기 → 설계 → AI에게 지시 → 결과 검증 → 수정 지시 → 테스트 → 배포
              ↑ 여기에 시간의 70%
```

차이가 보이시나요? **코드를 직접 타이핑하는 시간은 줄었지만, 설계와 검증에 쓰는 시간은 늘었습니다.**

이건 비유하면 이렇습니다:

- 예전: 벽돌을 하나하나 직접 쌓는 **벽돌공**
- 지금: 설계도를 그리고 로봇이 쌓은 결과를 확인하는 **건축가**

---

## 구체적으로 뭘 해야 하나?

AI 시대에 프론트엔드 개발자가 집중해야 할 역량을 4가지로 정리했습니다.

### 1. 설계 능력 — "뭘 만들지" 결정하는 힘

AI에게 "로그인 페이지 만들어줘"라고 하면 만들어줍니다. 그런데:

- 소셜 로그인을 넣을 건지?
- 비밀번호 찾기 플로우는?
- 세션 만료 시 어떻게 할 건지?
- 모바일에서 키보드가 올라오면 레이아웃은?

**이런 결정을 할 수 있는 사람이 진짜 개발자입니다.** AI는 "어떻게 만들까"는 알지만, "뭘 만들까"는 여러분이 정해줘야 해요.

### 2. 코드 리뷰 능력 — "이 코드가 맞는지" 판별하는 눈

AI가 코드를 빠르게 만들어주니까, 이제 중요한 건 **"이 코드를 쓸 수 있는가"**를 판단하는 능력이에요.

- 이 컴포넌트, 리렌더링 이슈는 없나?
- 접근성(a11y) 빠진 건 아닌가?
- 보안 취약점은?
- 프로젝트 컨벤션에 맞나?

GitHub의 개발자 경험(DevEx) 설문에서도, 개발자들은 **코드 품질이 코드 양보다 중요한 성과 지표**라고 답했습니다. AI가 양은 늘려주지만, 품질을 보장해주지는 않거든요.

> 📎 출처: [GitHub — Survey reveals AI's impact on the developer experience](https://github.blog/news-insights/research/survey-reveals-ais-impact-on-the-developer-experience/)

### 3. 소통 능력 — 기획자, 디자이너와의 다리

아이러니하게도, AI 시대에 가장 중요해진 역량 중 하나가 **사람과의 소통**입니다.

왜냐하면:

- 코드 짜는 시간이 줄었으니 → **논의하고 결정하는 시간이 늘어남**
- AI가 이해 못 하는 맥락을 → **사람이 전달해줘야 함**
- "디자이너가 의도한 인터랙션이 이거야" → AI에게 전달하려면 **내가 먼저 이해해야 함**

GitHub 설문에서 개발자들이 코드 품질 다음으로 중요하게 꼽은 성과 지표가 **"협업과 커뮤니케이션"**이었는데(33%만 실제 측정 중), 이건 앞으로 더 중요해질 거예요.

### 4. AI 도구 활용 능력 — 프롬프트 잘 쓰는 것도 실력

같은 Cursor를 써도, 결과물이 천차만별인 이유가 있어요.

- **잘 쓰는 사람**: "이 프로젝트는 Next.js 14 App Router를 쓰고, 상태관리는 Zustand야. UserProfile 컴포넌트를 만들어줘. API는 /api/users/:id고 반환 타입은 이거야."
- **못 쓰는 사람**: "프로필 페이지 만들어줘."

**좋은 프롬프트를 쓰려면, 기술을 알아야 합니다.** AI 도구를 잘 쓰는 것도 결국 개발 실력이에요. 공고에서 "Prompt Engineering"을 언급하는 이유가 여기에 있습니다.

원티드 프론트엔드 공고 중 메라키플레이스는 자격요건에 이렇게 썼어요:

> "AI 도구 마스터리: Cursor/Claude/Gemini/MCP 활용 + Prompt/Context Engineering"

프롬프트 엔지니어링이 프론트엔드 개발자 자격요건에 들어가는 시대입니다.

## "코딩 배울 필요 없는 거 아닌가요?"

이 질문에 대한 답은 **"아니요, 오히려 더 배워야 합니다"**입니다.

이유는 간단해요:

**1. AI가 만든 코드를 검증하려면 코딩을 알아야 합니다.**

AI가 아무리 잘해도 100% 완벽한 코드를 만들지는 못해요. Stack Overflow 설문에서도 개발자의 43%만 AI 출력을 신뢰한다고 답했고, 31%는 회의적이었습니다.

**2. AI에게 좋은 지시를 내리려면 기술을 알아야 합니다.**

"React로 만들어줘"와 "Next.js 14 App Router + Server Component로 만들어줘"는 결과물이 완전히 다릅니다. 기술을 알아야 구체적인 지시를 할 수 있어요.

**3. AI가 대체하는 건 "반복 작업"이지 "전문성"이 아닙니다.**

엑셀이 나왔다고 회계사가 사라지지 않았어요. 계산기를 눌러야 했던 시간이 줄었을 뿐, 판단은 여전히 사람의 몫이었죠. AI도 마찬가지입니다.

## 실제 공고에서 보이는 변화

원티드 프론트엔드 공고 101개에서, AI 시대의 역할 변화를 보여주는 문구들이 있었습니다:

> **플랭** — 자격요건
> "AI가 대신할 수 없는 설계와 판단을 고민하고, AI를 도구로 활용해 실행을 이끌 수 있는 사람"

이 한 문장이 이 글의 모든 내용을 요약합니다.

- "설계와 판단" — AI가 못하는 것
- "AI를 도구로 활용" — 개발자가 해야 하는 것
- "실행을 이끌 수 있는 사람" — 최종 결과물에 책임지는 건 사람

> **슈퍼센트** — 우대사항
> "기존 지식에 안주하지 않고, AI 발전 속도를 이해하고 자신의 역량으로 흡수할 수 있는 분"

AI를 안 써도 되는 게 아니라, **계속 따라가야 한다**는 뜻이에요.

## 신입에게 드리는 현실적인 조언

### 코딩 공부를 멈추지 마세요

AI 때문에 코딩을 안 배워도 된다고 하는 사람들이 있는데, 반대입니다. **AI가 만든 코드를 읽고 판단하려면**, 기초가 탄탄해야 해요.

### 하지만 공부 방식을 바꾸세요

- ❌ "이 코드를 외워야지" → ✅ "이 코드가 왜 이렇게 동작하는지 이해하자"
- ❌ "처음부터 끝까지 직접 짜야지" → ✅ "AI가 만든 걸 검증하고 개선하자"
- ❌ "구글에서 복붙하자" → ✅ "AI에게 맥락을 주고 더 나은 코드를 요청하자"

### 포트폴리오에 AI 활용 과정을 넣으세요

"AI 도구 없이 100시간 → AI와 함께 30시간" 같은 비교가 아니라,

> "Cursor + Claude를 활용해 초기 컴포넌트를 생성하고, 접근성 검증과 성능 최적화를 직접 수행했습니다"

이렇게 **"AI는 도구로 쓰고, 품질은 내가 책임졌다"**가 보이면 됩니다.

## 정리

| AI 시대 이전 | AI 시대 이후 |
|-------------|-------------|
| 코드를 직접 타이핑 | AI에게 지시하고 검증 |
| "얼마나 빨리 짜느냐" | "얼마나 좋은 코드인지 판별" |
| 구글/스택오버플로우 검색 능력 | 프롬프트 작성 능력 |
| 혼자서 묵묵히 개발 | 기획자·디자이너와 더 많이 소통 |
| 기술 스택 숙련도 | 기술 스택 이해도 + AI 활용 능력 |

**AI가 코드를 대신 짜주는 시대. 개발자는 "왜" 이렇게 만들어야 하는지 판단하는 사람이 됩니다.**

70%의 전문 개발자가 "AI는 내 직업에 위협이 아니다"라고 답한 이유도 여기에 있어요. 코드를 짜는 건 개발의 일부일 뿐, 설계하고 판단하고 소통하는 건 여전히 사람의 몫이니까요.
