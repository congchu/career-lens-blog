## 들어가며

프론트엔드 공부하다 보면 누군가 꼭 이렇게 말합니다.

> "Redux는 기본이지. 배워야 해."

2020년이었으면 맞는 말이었어요. 그런데 **2026년 채용공고를 보면, 현실은 완전히 달라졌습니다.**

---

## 공고 101개에서 상태관리 키워드를 세봤습니다

| 상태관리 도구 | 자격요건 등장 수 | 비율 |
|-------------|----------------|------|
| **Zustand** | 8개 | 8% |
| **Redux** | 7개 | 7% |
| **TanStack/React Query** | 11개 | 11% |
| **Recoil** | 5개 | 5% |
| **MobX** | 0개 | 0% |

**Zustand(8개)가 Redux(7개)를 넘었습니다.**

그리고 가장 많이 언급된 건 **React Query/TanStack(11개)**인데, 이건 엄밀히 말하면 "상태관리"가 아니라 "서버 상태 관리"예요. 이 차이가 중요합니다.

---

## "상태관리"의 의미가 바뀌었습니다

### 2020년: 모든 상태를 Redux에

```
Redux Store
├── user (로그인 정보)
├── products (API에서 받은 상품 목록)  ← 이것도?
├── cart (장바구니)
├── ui (모달, 사이드바 열림/닫힘)
└── loading (로딩 상태)               ← 이것도?
```

예전에는 **서버에서 받은 데이터(products)**도, **로딩 상태**도 전부 Redux에 넣었습니다. Action 만들고, Reducer 만들고, Dispatch 하고... 보일러플레이트가 엄청났죠.

### 2026년: 상태를 종류별로 분리

```
서버 상태 → React Query (자동 캐싱, 자동 리페치)
├── products, users, orders...

클라이언트 상태 → Zustand (간단, 보일러플레이트 거의 없음)
├── cart, ui, theme...

로컬 상태 → useState/useReducer
├── form inputs, toggles...
```

이렇게 바뀐 겁니다. 공고에서 React Query(11개)가 가장 많이 나오는 이유가 여기에 있어요. **서버 상태를 별도로 관리하는 게 이제 표준**이 됐거든요.

---

## Redux는 죽었나요?

**죽지는 않았지만, 신입이 첫 번째로 배울 도구는 아닙니다.**

| | Redux | Zustand |
|---|---|---|
| 공고 등장 | 7개 (7%) | 8개 (8%) |
| 보일러플레이트 | 많음 (Action, Reducer, Store) | 거의 없음 |
| 학습 곡선 | 가파름 | 완만 |
| 번들 크기 | ~10KB | ~1KB |
| 2026년 위치 | 레거시 유지보수 | 신규 프로젝트 선택 |

Redux가 공고에 나오는 7개 중 대부분은 **"기존에 Redux를 쓰고 있으니까"**입니다. 새 프로젝트를 Redux로 시작하는 곳은 거의 없어요.

> **Redux를 배울 시간에 Zustand + React Query를 배우세요.** 이게 2026년 현실입니다.

---

## 신입 추천 학습 순서

### 1단계: useState + useReducer (필수)

React 내장 훅만으로 로컬 상태를 관리하는 연습. 이게 기본기입니다.

```jsx
// 이것만으로도 충분한 경우가 많습니다
const [count, setCount] = useState(0);
const [items, dispatch] = useReducer(reducer, []);
```

### 2단계: React Query / TanStack Query (최우선)

서버에서 데이터를 가져오는 모든 곳에 사용. 공고 11개(11%)가 직접 언급하고, 실제로 쓰는 곳은 훨씬 많습니다.

**배울 것**: useQuery, useMutation, 캐싱, 낙관적 업데이트

### 3단계: Zustand (글로벌 상태가 필요할 때)

장바구니, 테마, 인증 상태 등 여러 컴포넌트에서 공유해야 하는 상태에 사용.

**배울 것**: create, middleware, persist

### 4단계: (선택) Redux Toolkit

Redux를 쓰는 회사에 지원할 때만. RTK(Redux Toolkit)로 배우면 보일러플레이트가 줄어듭니다.

---

## Recoil은요?

Recoil은 5개 공고에서 언급됐는데, 솔직히 말하면 **Meta가 공식 지원을 중단**했기 때문에 새로 배우는 건 추천하지 않습니다. 기존에 Recoil을 쓰는 회사가 있어서 공고에 나오는 것이고, 새 프로젝트에 선택하는 곳은 줄고 있어요.

Recoil 대신 **Jotai**를 쓰는 추세인데, 공고에서는 아직 Jotai 언급은 거의 없었습니다. 지금은 Zustand이 가장 안전한 선택이에요.

---

## 면접에서 상태관리 질문 나오면?

### "상태관리 도구 뭘 써봤나요?"

**좋은 답변**:
> "로컬 상태는 useState, 서버 상태는 React Query, 글로벌 상태는 Zustand로 분리해서 사용했습니다. 서버 상태를 별도로 관리하면 캐싱이 자동으로 되고, 로딩/에러 상태도 훅 하나로 처리할 수 있어서 코드가 훨씬 간결해졌어요."

**나쁜 답변**:
> "Redux를 썼습니다." (왜 Redux인지, 어떤 문제를 해결했는지 설명 없음)

### "Redux를 써봤나요?"

> "직접 프로젝트에 사용한 경험은 없지만, 개념(단방향 데이터 흐름, Action/Reducer 패턴)은 이해하고 있습니다. 제 프로젝트에서는 상태를 서버/클라이언트/로컬로 분리해서 Zustand + React Query를 선택했는데, Redux에 비해 보일러플레이트가 적어서 이 조합을 택했습니다."

이렇게 말하면 **"Redux를 모른다"가 아니라 "상태관리를 이해하고 적절한 선택을 했다"**로 들립니다.

## 정리: 2026년 상태관리 지도

**Redux 시대는 끝났습니다.** 2026년은 **"상태를 분류하고, 각각에 맞는 도구를 쓰는 시대"**입니다.
