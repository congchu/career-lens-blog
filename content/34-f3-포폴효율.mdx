## 들어가며

A씨는 3개월 동안 포트폴리오를 만들었습니다.
- 투두앱
- 날씨앱
- 영화 검색앱
- 개인 블로그

B씨는 2주 만에 포트폴리오를 만들었습니다.
- 실시간 데이터 대시보드 (1개)

**B씨가 먼저 합격했습니다.**

왜일까요? 시간의 문제가 아니라 **전략의 문제**입니다.

---

## 3개월 포폴이 지는 이유

### 이유 1: 프로젝트 수 ≠ 실력 증명

면접관은 포트폴리오 "개수"를 세지 않습니다.

| 3개월 포폴 (A씨) | 2주 포폴 (B씨) |
|------------------|---------------|
| 투두앱 | 실시간 대시보드 |
| 날씨앱 | |
| 영화 검색앱 | |
| 개인 블로그 | |
| **인상**: "튜토리얼 따라했구나" | **인상**: "이 사람은 실무 감각이 있네" |

4개의 간단한 프로젝트보다 **1개의 깊이 있는 프로젝트**가 훨씬 강력합니다.

### 이유 2: 기술적 도전이 없다

투두앱에서 쓰는 기술: useState, map, filter, localStorage
대시보드에서 쓰는 기술: React Query, WebSocket, 차트 라이브러리, 가상화, 무한 스크롤

면접관이 물어봤을 때:

> A씨: "useState로 상태를 관리했습니다"
> B씨: "WebSocket으로 실시간 데이터를 받는데, 초당 50건이 들어오니까 리렌더링이 심해져서 useMemo와 가상 리스트를 적용해서 해결했습니다"

**이야기의 깊이가 다릅니다.**

### 이유 3: "만든 이유"가 없다

"왜 이 프로젝트를 만들었나요?"

> A씨: "공부하려고요..." (튜토리얼 따라한 것)
> B씨: "커리어렌즈에서 채용 데이터를 실시간으로 모니터링하는 도구가 필요해서 만들었습니다"

**동기**가 있는 프로젝트는 면접에서 이야기가 풍부해집니다.

---

## 2주 만에 합격하는 포폴 만드는 법

### Step 1: 도메인 선택 (1일)

이전 편(C2)에서 분석한 공고 데이터를 다시 보면:

| 프로젝트 도메인 | 공고 비율 |
|----------------|----------|
| **대시보드/어드민** | 39% |
| **이커머스** | 23% |
| **실시간 기능** | 22% |

이 중 하나를 고르세요. **면접관이 "아, 실무에서 쓰겠다"고 느끼는 도메인**이어야 합니다.

### Step 2: 기술 스택 결정 (1일)

공고에서 가장 많이 요구하는 조합:

```
React 18 + TypeScript + React Query + Zustand
```

여기에 프로젝트 특성에 맞는 것 1~2개 추가:
- 대시보드 → 차트 라이브러리 (Recharts)
- 이커머스 → Next.js (SSR/ISR)
- 실시간 → WebSocket/SSE

### Step 3: 핵심 기능 구현 (7~10일)

**완벽한 앱이 아니라, "기술적 도전이 있는 핵심 기능"에 집중하세요.**

예시 - 대시보드 프로젝트:
- Day 1~2: 프로젝트 세팅 + API 연동 (React Query)
- Day 3~4: 테이블 컴포넌트 (정렬, 필터, 페이지네이션)
- Day 5~6: 차트 (실시간 업데이트)
- Day 7: 반응형 + 다크모드
- Day 8~9: 테스트 코드 1~2개 + 성능 최적화
- Day 10: 배포 + README

### Step 4: 차별화 포인트 만들기 (2~3일)

아래 중 **2개 이상** 적용하면 차별화됩니다:

| 차별화 요소 | 소요 시간 | 면접 효과 |
|------------|----------|----------|
| **테스트 코드** | 2~3시간 | "코드 품질에 관심 있구나" |
| **CI/CD** (GitHub Actions) | 30분 | "배포 자동화 할 줄 아네" |
| **성능 최적화** (Lighthouse) | 2~3시간 | "성능 감각이 있구나" |
| **에러 바운더리** | 1시간 | "에러 처리 신경 쓰네" |
| **접근성 (a11y)** | 2시간 | "시야가 넓구나" |

### Step 5: README 작성 (필수)

README가 포트폴리오의 **50%**입니다.

```markdown
# 프로젝트명

## 왜 만들었나?
(동기 — 개인적 필요 or 문제 해결)

## 기술 스택 + 선택 이유
- React Query: 서버 상태 캐싱 → API 호출 최소화
- Zustand: 클라이언트 상태 → 보일러플레이트 최소

## 핵심 기능
(스크린샷 또는 GIF)

## 기술적 도전 / 고민
- 문제: 대량 데이터 렌더링 시 60fps 유지
- 시도: react-window 가상화 적용
- 결과: 1000행 → 20행만 렌더링, FPS 55→60

## 개선할 점
(아직 부족한 점을 솔직히 → 성장 의지)
```

**"기술적 도전"과 "개선할 점"이 핵심입니다.** 면접관은 완벽한 프로젝트보다 **"문제를 인식하고 해결하려는 태도"**를 봅니다.

---

## 시간이 없을 때의 최소 전략

### 정말 1주일밖에 없다면

1. **기존 프로젝트 리팩토링** — 새로 만들지 말고, 있는 것을 개선
2. **TypeScript 마이그레이션** — JS → TS 전환만으로 어필 포인트
3. **테스트 코드 추가** — Jest/Vitest로 핵심 로직 테스트
4. **README 작성** — 기존 프로젝트에 README만 제대로 써도 인상 달라짐
5. **배포** — Vercel에 올리기 (3분)

### 가성비 최고 조합

> **기존 프로젝트 + TypeScript 전환 + 테스트 2개 + README + 배포**
> 
> 소요 시간: 3~5일
> 면접 효과: "JS 프로젝트를 TS로 마이그레이션한 경험" + "테스트 코드 작성" + "배포 경험"

이거면 신입 포트폴리오로 **충분히 차별화**됩니다.

---

## 면접관이 포트폴리오에서 보는 것 (우선순위)

1. **코드 구조** — 파일/폴더가 논리적으로 정리되어 있는가
2. **문제 해결 과정** — README나 커밋에서 고민의 흔적이 보이는가
3. **기술 선택 이유** — "왜 이 라이브러리를 썼는가" 설명할 수 있는가
4. **실행 가능** — 배포되어 있거나 clone 후 바로 돌아가는가
5. **완성도** — 기능이 어중간하지 않은가 (10개 반쪽보다 3개 완성)

**"몇 개 만들었나"는 5순위 밖입니다.**

---

## 정리

| 비효율적 (3개월) | 효율적 (2주) |
|----------------|------------|
| 프로젝트 4~5개 얕게 | 1~2개 깊이 있게 |
| 튜토리얼 클론 | 실무 도메인 선택 |
| useState만 사용 | React Query + 상태 분리 |
| README 없음 | README에 기술적 도전 기록 |
| 로컬에서만 실행 | 배포 + CI/CD |
| "공부하려고 만들었어요" | "이 문제를 해결하려고 만들었어요" |

**3개월을 쓰든 2주를 쓰든, 결국 "깊이"가 다릅니다.** 시간이 없다고 포기하지 마세요. 전략적으로 집중하면 2주로도 충분합니다.
